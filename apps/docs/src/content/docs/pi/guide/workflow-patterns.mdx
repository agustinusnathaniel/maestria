# Workflow Patterns

The 7 specialist subagents work as a team. The orchestrator manages the flow — it decomposes your request into phases, delegates to specialists via `maestria_subagent`, and enforces phase gates before each stage begins.

## Explore a codebase

Send `@adventurer` to map unfamiliar code before making changes.

> `@adventurer` Map the API layer — find all routes, middleware, and error handlers.

The adventurer traces call chains, maps modules, and reports back. Start here whenever you edit code you haven't worked with before.

## Make a design decision

Delegate to `@architect` when you need to evaluate options.

> `@architect` Should we use REST or GraphQL for the new public API? Evaluate trade-offs.

The architect returns a trade-off analysis with a clear recommendation.

## Plan a feature

Get a phased plan from `@planner` before implementing multi-step features.

> `@planner` Plan the implementation for a multi-tenant organization feature.

The planner breaks work into ordered milestones with verification criteria.

## Build something

Start with a mode command and let the spec-driven pipeline run.

> `/fein` We need a WebSocket-based notification system.

The orchestrator decomposes the task into a spec, then sequences through specialists: adventurer → architect → planner → builder → reviewer. Each phase waits for a completion gate before the next begins. Code changes always go through builder; every review through reviewer.

## Fix a bug

Start with `@diagnose` to find the root cause, then hand the fix to the orchestrator.

> `@diagnose` Users are getting a 503 error when uploading files over 10MB. Find the root cause.
>
> `/fein` Fix the upload issue that @diagnose identified.

The diagnose agent runs a 6-step tracing loop and reports the root cause with a suggested fix.

## Review code

Let `@reviewer` check your changes before committing.

> `@reviewer` Review my current changes before I commit.

The reviewer runs a 7-dimension checklist: correctness, performance, security, maintainability, test coverage, error handling, and style.

## Document something

Delegate READMEs, ADRs, changelogs, and API docs to `@writer`.

> `@writer` Write a README for the new caching package.

The writer follows structured patterns matched to the document type. Description in, complete document out.

## Parallel exploration

Fan out multiple adventurers to understand several areas at once.

> `/fein` I need to understand both the auth system and the payment system. Map them in parallel.

The orchestrator spawns independent tasks simultaneously and presents a combined report.

## Pipeline reference

```
@orchestrator (manager)
  ├─ Spec generation     → decompose request into phases
  ├─ @adventurer         → Reconnaissance
  ├─ @architect          → Design/decisions
  ├─ @planner            → Phased plan
  ├─ @builder            → Implementation
  └─ @reviewer           → Validation

Subagent dispatch: maestria_subagent (via @gotgenes/pi-subagents)
Phase gates: completion check before each stage transition
```

Specialists are invoked via `maestria_subagent`. The built-in `explore` and `general` agents are not part of this pipeline.

## Workflow Modes

Mode commands tell the orchestrator how to handle your request. They're slash commands that work like Pi's built-in command system:

| Command  | Effect                                                              | When to use                                          |
| -------- | ------------------------------------------------------------------- | ---------------------------------------------------- |
| `/fein`  | Full pipeline: recon → design → build → review with mandatory gates | Production-grade changes where quality matters       |
| `/sonar` | Research only: recon + design, then STOP before implementation      | Exploring unfamiliar code, evaluating approaches     |
| `/blitz` | Fast implementation: builder direct, skip recon and review          | Quick fixes, prototypes, code you already understand |

Use a mode command at the start of your message, with or without a task description:

```bash
/fein Add a rate limiter
/sonar Trace the auth flow
/blitz Fix this typo in the error message
```

When no mode command is given, the orchestrator uses its default pipeline. If multiple mode commands appear, the most restrictive mode wins: `/fein` > `/sonar` > `/blitz`.

<Aside type="tip">
  Use <code>/fein</code> for anything that touches multiple files or crosses module boundaries. Use{' '}
  <code>/sonar</code> when you need to understand something before committing to a solution. Use{' '}
  <code>/blitz</code> when you know exactly what to do and just need it done.
</Aside>

<Aside type="tip">
  Most tasks don't start with code. Start with recon (@adventurer) or design (@architect) before
  reaching for @builder. Mode commands handle this routing automatically.
</Aside>
