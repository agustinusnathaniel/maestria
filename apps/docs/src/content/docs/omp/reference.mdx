## Plugin Detection

OMP v17 stores npm packages at `~/.omp/plugins/node_modules/` — this differs from Pi's `~/.pi/agent/npm/` convention. Detection involves three checks:

| Check            | Mechanism                                                             |
| ---------------- | --------------------------------------------------------------------- |
| **CLI presence** | `commandExists('omp')` — verifies the `omp` CLI binary is on `$PATH`  |
| **Installation** | Existence of `~/.omp/plugins/node_modules/@maestria/omp/package.json` |
| **Version**      | Reads version field from the same `package.json`                      |

The three checks are independent: CLI may be installed without the plugin, and vice versa. Version detection only fires when the plugin is installed.

## Install & Update Mechanics

Unlike Pi, which manages its npm package directly, OMP delegates all package operations to the `omp` CLI:

| Operation             | Command                               |
| --------------------- | ------------------------------------- |
| **Install**           | `omp install @maestria/omp`           |
| **Update to version** | `omp install @maestria/omp@<version>` |

The `omp` CLI automatically resolves the `@scope/name` format against its npm registry and uses `bun install` internally for dependency management. No manual `npm` or `bun` commands are needed in the plugin directory.

## Known Issues

### npm: prefix self-alias loop

Using `npm:@maestria/omp` instead of the bare `@maestria/omp` triggers a bun dependency loop.

**Error signature:**

```
error: Package "@maestria/omp@0.2.3" has a dependency loop
```

**Root cause:** When bun encounters an `npm:` prefixed dependency, it stores it as an alias in `~/.omp/plugins/package.json`:

```json
"@maestria/omp": "npm:@maestria/omp"
```

Because the alias key equals the resolved package name, bun creates a self-referencing cycle: the alias points to itself, which bun tries to resolve, which points to the alias again.

**Fix:** Always use bare package names — never the `npm:` prefix.

**Remediation for an already-broken install:**

```bash
cd ~/.omp/plugins && bun install @maestria/omp
```

This replaces the stale `npm:` alias entry with a normal semver range, breaking the cycle.

## Lifecycle

Install, update, and uninstall all delegate to the `omp` CLI rather than managing plugin files directly. Unlike Pi, OMP has no subagent prerequisite — no equivalent of Pi's `@gotgenes/pi-subagents` requirement. The `omp` CLI is the sole interface for plugin lifecycle management.
