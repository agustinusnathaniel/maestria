`@maestria/opencode` provides one manager agent (`@orchestrator`) and 7 specialist subagents. Each agent has a focused domain, clear permissions, and a specific place in the pipeline.

## Orchestration Model

The `@orchestrator` is the manager. It decomposes complex tasks into atomic units, delegates each unit to the right specialist, integrates the results, and verifies completion. Specialists each handle one domain — exploration, design, implementation, debugging, planning, review, or documentation.

The default pipeline for non-trivial work follows a sequence: recon first, then design or plan, then build, then review. Each step verifies before handing off to the next.

## Pipeline Flow

```
@orchestrator (manager)
  ├─ @adventurer → Reconnaissance
  ├─ @architect  → Design/decisions
  ├─ @planner    → Phased plans
  ├─ @builder    → Implementation
  └─ @reviewer   → Validation
```

## Workflow Modes

The orchestrator responds to three mode keywords that adjust the pipeline depth:

| Keyword | Full form           | Pipeline                                            | Use case                           |
| ------- | ------------------- | --------------------------------------------------- | ---------------------------------- |
| `fein`  | Full pipeline       | adventurer → architect/planner → builder → reviewer | Production-grade, mandatory review |
| `sonar` | Research            | adventurer → architect/planner → STOP               | Understanding before committing    |
| `blitz` | Fast implementation | builder direct                                      | Quick fixes, known territory       |

The keyword is detected at any position in your message, case-insensitively. Most restrictive keyword wins (fein > sonar > blitz). Code blocks are excluded from detection.

To disable a mode, set `modes.disabledKeywords` in your plugin configuration:

```jsonc
{
  "plugin": ["@maestria/opencode", { "modes": { "disabledKeywords": ["blitz"] } }],
}
```

## Agent Reference

| ----------------- | ---------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| **@orchestrator** | `all`      | Manager — decomposes work, delegates, never implements                               | Complex multi-step tasks, cross-agent workflows, anything needing structure — supports `fein`/`sonar`/`blitz` workflow modes | `edit: deny`, task delegates to 7 specialists only |
| **@adventurer**   | `subagent` | Codebase reconnaissance — maps unfamiliar code, traces dependencies, finds key files | Before implementing in unfamiliar code, understanding how something works, finding where code lives                          | `edit: deny`, read/search only                     |
| **@architect**    | `subagent` | Architecture decisions — evaluates options with trade-off analysis, writes ADRs      | Choosing between technologies, making design decisions, evaluating approaches                                                | `edit: deny`, `webfetch: allow`                    |
| **@builder**      | `subagent` | Focused implementation — one atomic task per invocation                              | Bug fixes, feature work, tests, refactors — after recon and design are done                                                  | `edit: allow`, `bash: * ask`                       |
| **@diagnose**     | `subagent` | Systematic debugging — traces errors from symptom to root cause                      | Bugs, regressions, crashes, mysterious errors, failing tests                                                                 | `edit: ask`, `bash: * ask`                         |
| **@planner**      | `subagent` | Implementation plans — phased milestones with verification                           | Complex features needing structured rollout, migration plans, multi-phase work                                               | `edit: ask`, `webfetch: allow`                     |
| **@reviewer**     | `subagent` | Code review — correctness, edge cases, security, performance, maintainability        | Pre-merge review, security audit, post-implementation validation                                                             | `edit: deny`                                       |
| **@writer**       | `subagent` | Documentation — structured patterns for READMEs, API docs, changelogs, ADRs          | Documenting code, writing READMEs, API docs, changelogs, ADRs                                                                | `edit: allow`                                      |

## Usage Notes

**Direct delegation.** Any agent can be invoked directly by typing `@agentname` in your prompt. For example, `@reviewer review my latest changes` or `@adventurer find where authentication is handled`.

**Manager delegation.** For complex tasks involving multiple steps, let `@orchestrator` handle the entire pipeline. It decomposes the work and dispatches each piece to the right specialist automatically.

**Pipeline discipline.** The default flow for non-trivial work is adventurer (recon) → architect or planner (design/plan) → builder (implement) → reviewer (validate). Each step verifies before handing off. Skipping recon or review trades speed for risk — fine for simple changes, not recommended for cross-module work.
